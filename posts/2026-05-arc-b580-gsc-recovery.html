Intel Arc B580 firmware debugging (GSC recovery)

Notes from debugging a Linux bring-up failure involving Intel Arc B580 (BMG G21 / Battlemage)

Spent the day trying to untangle firmware weirdness on my Arc B580.

Arc seems to split responsibilities across multiple firmware layers: PCode, GSC, OpROM code/data, GuC, and HuC. If one layer fails early enough, the entire graphics stack can die before /dev/dri even exists. (New GPU firmware stacks are annoyingly complicated, but Intel exposing these layers publicly made debugging possible.)

In my case, it looked like the previous owner corrupted the GSC firmware somehow. Every cold boot caused the xe driver to fail during early initialization while interacting with PCode, so the card never came online properly.

Debugging path

  • Intermittent /dev/dri creation
  • LightDM crashing with “no screens found”
  • PCode initialization timeouts
  • “Survivability mode”
  • Silent xe hangs immediately after “deactivate VGA console”
  • Different kernel behavior (6.8, 6.17 generic, 6.17 OEM)
  • Card occasionally disappearing from lspci after reseating

At one point it looked more like hardware damage than firmware corruption. I checked for obvious shorts on 12V and 3.3V rails, and measured resistance/voltages on Vcore, memory, and memory controller rails. Nothing abnormal showed up, but that’s the extent of my EE knowledge.

Recovery

Using Intel’s igsc tool compiled from source, I was able to access the Graphics Security Controller through /dev/mei1 and reflash the GSC firmware from 21.1137 to 21.1182.

After firmware update

  • Mesa detected: Intel Arc B580 Graphics (BMG G21)
  • Vulkan initialized normally
  • No more LLVM software rendering for OpenGL

Remaining issues

There appears to be a separate upstream issue in Linux 6.17 xe, related to GuC scheduler idle crashes on Battlemage. This seems unrelated to the firmware issue.

Linux support for Arc has come a long way. The combination of xe, Mesa, fwupd, and tools like igsc made recovery possible without needing to immediately switch OS.

That said, firmware distribution is still fragmented. My ASRock card firmware is not available on LVFS yet, so full recovery may still require vendor flashing tools on Windows for some regions.

Linux graphics on Intel Arc is still a surprisingly deep stack.
